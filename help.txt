===============================================================================
 ZKas ($zkas) — Mining Pool Operator Guide
===============================================================================

This document explains what the ZKas chain is, how to point an existing
Kaspa-style pool at it, and how ZKas's dual-acceptance merged mining works
(before and after activation). It is written for pool operators who already run
a kHeavyHash (Kaspa) pool and want to add ZKas.


-------------------------------------------------------------------------------
 1. WHAT FIRECASH IS
-------------------------------------------------------------------------------

  - A fork of rusty-kaspa (kaspad v2.0.1 lineage). Same GHOSTDAG / BlockDAG
    consensus, same UTXO base layer, same RPC surface.
  - Privacy: shielded-by-default. Value lives in an Orchard (Halo 2, zk-SNARK)
    shielded pool; the coinbase reward is minted directly as a shielded note.
    This does NOT change mining — you still mine plain block headers.
  - Proof of Work: kHeavyHash, BYTE-IDENTICAL to Kaspa mainnet. The PoW files
    are an untouched fork, so any Kaspa miner / ASIC / hashing library that
    solves Kaspa headers solves ZKas headers with zero changes.
  - Target block rate: 1 BPS. Coinbase emission: 60 $zkas per block (at 1 BPS),
    i.e. 6e9 sompi/block (1 $zkas = 1e8 sompi). The initial reward halves every
    3 months, caught by a two-step perpetual tail (6 -> 3 $zkas/block). Ticker:
    $zkas.
  - Address format: bech32 with HRP "zkas:" (mainnet), e.g.
    zkas:pyfjy228...  (testnet "zkastest:", devnet "zkasdev:",
    simnet "zkassim:"; legacy firecash* prefixes stay valid). Network id / datadir name stays "firecash-mainnet" (wire/disk identity, not display).
  - Default ports: node gRPC 16110, P2P 16111 (same numbering as Kaspa's
    16110/16111 — run ZKas on its own host or remap if co-locating).

  Because the PoW is identical to Kaspa's, standing up a ZKas pool is
  exactly like standing up a Kaspa pool: get a block template from a ZKas
  node, hand it to miners, submit solved blocks back. Nothing about the shielded
  layer touches the stratum path.


-------------------------------------------------------------------------------
 2. THE TWO SOLUTION TYPES FIRECASH ACCEPTS
-------------------------------------------------------------------------------

A ZKas block is valid if EITHER of these holds:

  (A) NATIVE solution
      The block's own kHeavyHash header clears the block's ZKas target.
      This is ordinary mining — identical to Kaspa. It is the BACKBONE of the
      chain: liveness never depends on Kaspa miners. Native solutions are
      accepted at all times, before and after merged-mining activation.

  (B) AUXPOW (merged-mined) solution
      The block carries an AuxPoW proof: a PARENT kHeavyHash block (typically a
      Kaspa block) whose work clears *ZKas's own* target and which is
      cryptographically bound to this ZKas block. Accepted ONLY once merged
      mining is active (see section 4).

Key point on difficulty: the AuxPoW is checked against FIRECASH'S OWN target
(its own DAA retarget), NOT Kaspa's. A hash that fails Kaspa's hard target
routinely clears ZKas's easier one, so merged mining augments ZKas
security at essentially zero marginal cost to a Kaspa miner.

The verifier always tries the NATIVE check first, then the aux check. A valid
native block can never be invalidated by a malformed or stripped aux proof,
because the aux data is excluded from the block hash (see section 5).


-------------------------------------------------------------------------------
 3. ENABLING NATIVE MINING (any Kaspa-style pool works today)
-------------------------------------------------------------------------------

No special support needed — treat it like a Kaspa pool:

  1. Run a ZKas node:
       kaspad --appdir=<dir> --rpclisten=127.0.0.1:16110 --utxoindex
     (add --enable-unsynced-mining only for the first/solo node bootstrapping a
      brand-new network with no peers yet).

  2. Point your stratum bridge at it (kaspad_address: 127.0.0.1:16110).
     get_block_template / submit_block behave exactly as on Kaspa. kHeavyHash
     ASICs (Bitmain, IceRiver, Goldshell, etc.) work unmodified.

  3. Pay miners to a zkas: address. (In this reference pool the reward is a
     shielded coinbase note; payout accounting is handled by the pool, not the
     miner — the miner only ever sees plain headers.)

This path is fully live and is what the network runs on.


-------------------------------------------------------------------------------
 4. MERGED-MINING ACTIVATION TIMELINE (before / after)
-------------------------------------------------------------------------------

Consensus supports GATING merged mining behind a fork activation keyed on DAA
score. The mechanism is a `ForkActivation`: before the activation score only
native solutions are accepted; at/after it, both native and AuxPoW are accepted.

    CURRENT live network: merged mining is active from GENESIS
                          (merged_mining_activation = always).
    A production relaunch MAY instead gate it to a DAA score (e.g. a ~2-week
    delay = 1,209,600 DAA at 1 BPS), flipping to the BEFORE/AFTER behaviour below.

  BEFORE activation (if gated):
      - ONLY native solutions (type A) are accepted.
      - AuxPoW proofs on a header are ignored for PoW purposes.
      - Every pool mines ZKas natively. Nothing to configure.

  AT/AFTER activation (and on the current always-on network):
      - BOTH native (A) and AuxPoW (B) solutions are accepted.
      - Pools MAY switch to merged mining with Kaspa, or keep mining natively —
        both remain valid forever. There is no flag day for miners; native is
        never turned off.

The gate is evaluated on the block's own DAA score, which consensus re-derives
and enforces in context, so a block cannot "lie" about its score to unlock aux
acceptance early.


-------------------------------------------------------------------------------
 5. THE AUXPOW PROTOCOL (how a merged solution is built and verified)
-------------------------------------------------------------------------------

Binding chain (what the verifier walks):

    pow(parent_header) -> parent_header.hash_merkle_root -> parent_coinbase -> H_fc

  where H_fc is the ZKas block hash. To merge-mine one ZKas block:

  1. Build the ZKas block and compute its hash H_fc. NOTE: H_fc is computed
     over the explicit header fields and EXCLUDES the aux_pow data, so it is
     stable no matter what parent later carries it.

  2. Embed the commitment in the PARENT block's coinbase. The parent coinbase
     payload (the miner-writable extra_data slot — the only field you can set in
     a real Kaspa block without changing Kaspa) must contain, exactly once:

         MERGE_MINE_MAGIC (4 bytes, ASCII "FCMM") || H_fc (32 bytes)

     "Exactly once" is enforced (ambiguous/duplicate commitments are rejected,
     erring safe).

  3. Mine the parent header with kHeavyHash against FIRECASH'S target (parent
     bits = ZKas's bits). Because that target is easier than Kaspa's, most
     Kaspa-failing hashes still qualify here.

  4. Submit the ZKas block with an AuxPow proof attached:

         AuxPow {
             parent_header:         Header,        // the mined parent
             parent_coinbase:       Transaction,   // carries FCMM||H_fc
             coinbase_merkle_branch: Vec<Hash>,    // coinbase (leaf 0) -> root
         }

  Verifier (verify_aux_pow) checks:
     a. committed_hash(parent_coinbase) == H_fc                 (binding)
     b. fold(coinbase, branch) == parent_header.hash_merkle_root(inclusion)
     c. kHeavyHash(parent_header, parent.nonce) <= ZKas target (work)

  The parent need NOT be a valid Kaspa block — only that real kHeavyHash work,
  bound to H_fc, cleared ZKas's target. Stealing another block's PoW fails:
  a parent committing to fc_a cannot be reused for fc_b (commitment mismatch).


-------------------------------------------------------------------------------
 6. WIRING MERGED MINING IN THIS POOL
-------------------------------------------------------------------------------

This reference pool (fork of Nacho-the-Kat/katpool) ships an opt-in merged path:

    FIRECASH_MERGED_MINING=1   (env on the stratum bridge)

  When set, the bridge decorates the node RPC (bridge/src/merged.rs,
  bridge/src/kaspaapi.rs):
    - get_block_template hands the ASIC a PARENT block that already commits to
      the current ZKas H_fc, with parent bits = ZKas's target.
    - submit_block turns a solved parent back into a ZKas aux block and
      submits it (the aux proof rides RpcRawHeader.aux_pow, borsh-hex encoded).
  The stratum/job/share core is untouched, so the native ASIC path cannot
  regress. Miners connect and hash exactly as before.

  IMPORTANT CAVEAT — synthetic vs. real Kaspa parent:
  Out of the box the "parent" the bridge builds is SYNTHETIC (ZKas-only).
  That proves the aux path end to end, but the miner earns only ZKas — there
  is no Kaspa reward, because the parent is not submitted to Kaspa. To get TRUE
  dual-chain merged mining (Kaspa + ZKas from one hash):
     (a) pull the parent template from a real Kaspa node, insert FCMM||H_fc into
         its coinbase extra_data, and submit winning parents to BOTH Kaspa (as a
         Kaspa block) and ZKas (as an aux block);
     (b) make the payout accountant track the parent hash in merged mode.
  Until (a)+(b) are done, run merged mode for validation only, not for revenue.


-------------------------------------------------------------------------------
 7. RPC / WIRE REFERENCE (for custom pool software)
-------------------------------------------------------------------------------

  - Header field:   Header.aux_pow: Option<Box<AuxPow>>  (borsh; None = native).
                    Excluded from hashing::header::hash, so H_fc is unchanged.
  - P2P:            BlockHeader.auxPow = field 15 (p2p.proto), borsh bytes;
                    empty bytes == native header.
  - gRPC RPC:       RpcBlockHeader.auxPow = field 16; RpcRawHeader.aux_pow is a
                    borsh-hex string, so submit_block over the wRPC/workflow path
                    carries the aux proof.
  - CAVEAT:         the plain gRPC RpcBlockHeader->Header conversion currently
                    sets aux_pow = None (drops aux). Submit merged blocks via the
                    RpcRawHeader / wRPC path, not the bare gRPC header path.
  - Constants:      MERGE_MINE_MAGIC = b"FCMM". Commitment = the 32-byte H_fc.

  Source of truth:
    consensus/core/src/auxpow.rs   — AuxPow type, commitment scan, inclusion
    consensus/pow/src/auxpow.rs    — verify_aux_pow, check_pow_gated (the gate)
    (rusty-kaspa fork; the pool consumes these via its patched kaspa-* deps)


-------------------------------------------------------------------------------
 8. PAYOUT MODEL — VESTING, AUTO-SEND, AND EARLY CLAIM
-------------------------------------------------------------------------------

Because every coinbase reward is a SHIELDED note (no transparent UTXO a miner
can be handed directly), the pool credits each matured reward to the miner's
balance and later sends it from the pool's shielded treasury. How and when it
leaves the pool is governed by a single 10-day cliff:

  VESTING CLIFF = 10 days (accountant/src/vesting.rs: VESTING_CLIFF_DAYS).

  Each individual reward vests on its own age since it matured:

    - AGE >= 10 days  ->  pays 100%  (FULL_PAYOUT_BPS  = 10000)
    - AGE <  10 days  ->  pays  50%  (EARLY_PAYOUT_BPS =  5000)

  The split steps at the cliff (no linear ramp). Arithmetic is integer sompi
  with an i128 intermediate, so miner + forfeit == gross exactly — the split
  can never create or destroy value.

TWO WAYS MONEY LEAVES THE POOL (accountant/src/payout.rs: PayoutTrigger):

  (1) AUTO-SEND  (PayoutTrigger::AutoSweep)  — the default, no action needed.
      A scheduled sweep picks up every reward that has passed the 10-day cliff
      and sends it to the miner's zkas: address at 100%. NO SIGNATURE is
      required: the funds are fully vested and go to the address the miner
      already mined to. Rewards still inside the cliff are left to keep vesting
      and are picked up by a later sweep once they mature.

  (2) EARLY CLAIM (PayoutTrigger::SignedClaim) — optional, miner-initiated.
      A miner who wants their balance BEFORE the cliff signs a one-time,
      address-bound challenge with their shielded key (api/src/claim.rs). This
      pays everything now: matured rewards at 100% and still-vesting rewards at
      50%. The 50% haircut is the price of early liquidity; the withheld half is
      routed per ForfeitPolicy (default: pool treasury).

  In one line: SIGNING IS ONLY EVER NEEDED TO CLAIM EARLY (before 10 days). After
  10 days a reward auto-sends at full value with nothing for the miner to do. A
  signature is a miner's choice to trade 50% for speed — never a requirement to
  eventually get paid.

  Note: a fully-matured reward pays 100% on BOTH paths, so there is never a
  reason to sign for money you can get free at full value on the next sweep.


-------------------------------------------------------------------------------
 9. SUMMARY FOR A POOL OPERATOR
-------------------------------------------------------------------------------

  - Want to mine ZKas today?  Run a ZKas node + your Kaspa-style bridge.
    It's kHeavyHash — nothing special. Native solutions are always valid.
  - On the current network merged mining is active from genesis, so you MAY
    additionally submit merged (AuxPoW) solutions built on a Kaspa parent; native
    keeps working unchanged. (If a future relaunch gates it to a DAA score, aux is
    accepted only at/after that score.)
  - For real Kaspa+ZKas dual rewards you must feed a real Kaspa template and
    submit winning parents to both chains (section 6 caveat).
